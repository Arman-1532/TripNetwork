WHY CHOOSE MYSQL FOR THIS PROJECT? (RELATIONAL VS. NOSQL)

Use this script to explain your architectural choice clearly and professionally.

---

1. DATA INTEGRITY & ACID COMPLIANCE
"The biggest reason for choosing MySQL was data integrity. TripNetwork handles financial transactions (payments) and critical bookings. In a relational database like MySQL, we get ACID (Atomicity, Consistency, Isolation, Durability) by default. This ensures that if a user pays, the system either completes the entire transaction (updating the payment AND the booking) or nothing at all. In a NoSQL database like MongoDB, achieving this same level of reliability across multiple collections is much more complex and often suffers from 'eventual consistency' issues."

2. RELATIONAL DATA STRUCTURE
"Our data is highly connected. For example:
   - A Booking belongs to a User.
   - A Booking references a Package.
   - A Package is owned by a Provider (Agency).
MySQL is designed specifically to handle these relationships using JOIN operations. This allows us to fetch all the related data in a single, efficient query. In NoSQL, we would have to either duplicate (denormalize) data—which leads to messy updates—or do multiple manual lookups in the application code, which is slower."

3. SCHEMA ENFORCEMENT & QUALITY
"In a travel application, certain fields are absolutely non-negotiable. A booking *must* have a price, a date, and a user ID. MySQL enforces a strict schema, meaning the database itself will reject any 'bad data' before it ever reaches our application logic. This prevents 'silent bugs' that could crash the UI later on."

4. THE "HYBRID" ADVANTAGE (Pro Point)
"Interestingly, I didn't lose the flexibility of NoSQL. I used MySQL's **JSON support** for the bidding logic. This gave me the best of both worlds: I have the rock-solid RELATIONAL structure for the core system (Payments/Users) while keeping the DOCUMENT-like flexibility for the dynamic bids. It's a more modern and robust architecture than using a purely non-relational database for everything."
